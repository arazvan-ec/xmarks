## What changed

- New `upgrades/` system: every release now ships an AI-authored note like this one, and
  `/flywheel:update` executes the strategies it finds in the update range.
- The updating AI logs applied strategies to `.claude/flywheel/UPGRADES.md` in the target
  repo (created on first use).
- Carried over from 0.7.0 for repos that skipped it: install manifest + `.pre-flywheel`
  backups, session-start new-version notice, optional weekly auto-update PR workflow
  (`--auto-update`).

## Strategy

1. **Verify the refresh brought the manifest** — check `.claude/flywheel/.manifest`
   exists. It appears automatically after re-running the installer (any install ≥ 0.7.0);
   if missing, the refresh didn't run — re-run `install-vendored.sh` before continuing.
2. **Offer auto-update PRs if absent** — if `.github/workflows/flywheel-update.yml` does
   not exist, ask the user whether they want weekly auto-update PRs; if yes, re-run the
   installer with `--auto-update`. Then derive `<owner>/<repo>` from `git remote get-url
   origin` and hand the user the exact toggle URL —
   `https://github.com/<owner>/<repo>/settings/actions` → *Workflow permissions →
   "Allow GitHub Actions to create and approve pull requests"* — that setting cannot be
   automated; the user must flip it in the GitHub UI.
3. **Surface overwritten local edits** — list any `*.pre-flywheel` files under `.claude/`;
   if there are any, show them to the user so hand-edits aren't silently lost.
