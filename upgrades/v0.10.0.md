## What changed

- **Typed ledger entries.** `/flywheel:compound` now writes each learning as
  `## <type>: <title>` followed by a single greppable `<!-- fw: type=…; date=…;
  files=…; spec=…; pr=…; branch=… -->` metadata line, then the prose body.
  `type` and `date` are required; `files`/`spec`/`pr`/`branch` are filled in
  when known.
- **Relevance-based, budgeted injection.** `scripts/session-start.sh` no longer
  dumps a blind ~50-line head of `.claude/flywheel/LEARNINGS.md`. It now scores
  every entry against the current git context (`+3` overlapping `files`, `+2`
  matching `branch`, `+2` matching the active spec, `+1` recency) and injects
  the top `N` full entries (default `N=12`, override with
  `FLYWHEEL_LEARNINGS_INJECT`), followed by a one-line pointer for the rest
  (`… N more learnings — run /flywheel:recall <query> to pull specifics.`).
- **New `/flywheel:recall <query>`** — greps entry titles/metadata for a cheap,
  numbered list of matches, then expands one entry's full body on request.
  Added to the README command table and `/flywheel:help`.
- **Fully backward-compatible.** Old free-prose entries (no metadata comment)
  still load — they're treated as always-eligible, low-priority (score 0) —
  and a ledger with no `## ` headers at all falls back to the previous raw
  head-dump behavior. The hook's contract is unchanged: read-only, idempotent,
  fail-open, always exits 0 (including when there is no ledger or the project
  isn't a git repo).
- No new binary or dependency — `git`, `awk`, and `grep` only. No config
  migration: existing ledgers work as-is, and new entries adopt the typed
  format going forward. Vendored and marketplace installs pick this up on the
  next re-vendor / reload — no action required in installed repos.
