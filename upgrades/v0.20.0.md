## What changed

- **Trust boundary on the gate (the headline fix).** `scripts/gate.sh` (the
  `Stop` hook) used to execute `.claude/flywheel/gate.sh` automatically. That
  file lives in the repo, so any PR or clone could add or rewrite it (the
  executable bit survives checkout) and have it run on the victim's machine at
  the next turn end — code execution by PR. Now an **unrecognized gate is not
  executed**: the hook prints the one command that trusts it — a content hash
  recorded **outside the repo tree**
  (`${XDG_STATE_HOME:-~/.local/state}/flywheel/trusted-gates`), so a PR can't
  self-authorize — and lets the turn end. Editing the gate changes its hash and
  revokes trust until you re-consent. Trust is **fail-safe** (an unverified gate
  never runs); every other path stays **fail-open** (any internal error is a
  no-op, never a block).
- **Cost: no redundant suite runs.** The gate skips running when the
  **git-tracked** working tree is byte-for-byte the state that last passed (a
  signature over `HEAD` + `git status` + `git diff HEAD` + untracked file
  content, cached on green), so no-change turns and the moment right after
  `/flywheel:verify` don't re-run the whole suite. It is a cache, never a gate:
  any git-visible change (staged, unstaged, or untracked) re-runs, and a
  non-git project always runs. Non-git state (env vars, git-ignored files) is
  not observed — the cache only ever *skips* a re-run, never turns a red gate
  green for a changed tree.
- **Escape valve that persists, keyed per failing tree.** After the bounded MAX
  consecutive blocks the bypass is recorded against that failing tree's
  signature (cleared only by a green run), so a permanently-red gate no longer
  re-traps every later turn — and the block counter is keyed to the failing
  signature, so one bypass never disables enforcement for a *different* later
  regression. The hook also honors `stop_hook_active` from its stdin as
  defense-in-depth.
- **First test coverage.** New `scripts/test-gate.sh` (wired into CI) exercises
  every path — including proving an untrusted gate is never executed via a
  sentinel — the coverage the gate never had.

## Strategy

Applied per repo that has **already opted into the gate** (has an executable
`.claude/flywheel/gate.sh`). Repos without one need no action.

1. After updating, the next time the `Stop` hook fires it will print, on stderr,
   that the gate is not trusted yet plus the exact command to trust it —
   something like:
   `mkdir -p "$HOME/.local/state/flywheel" && echo <hash> >> "$HOME/.local/state/flywheel/trusted-gates"`
2. **Review `.claude/flywheel/gate.sh` first** (confirm it only runs your
   intended verification command), then run that printed command once.
3. The gate resumes blocking on red as before. If you later edit the gate, trust
   is revoked and step 1 repeats — re-run the (new) printed command.
4. Verify: with the gate green, finishing a turn is allowed; introduce a
   deliberate failure and confirm the turn is blocked. No migration touches the
   repo — the consent lives in your user state dir, not the project.
