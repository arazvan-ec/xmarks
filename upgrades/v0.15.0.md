## What changed

- **New pillar: an agent-native runtime.** Where the existing loop *builds*
  software, this lets flywheel *operate* it — Claude runs recurring domain
  operations ("analyze a car", "score a lead") as the backend, following a fixed
  contract, instead of a static code service. Direction and full design:
  `docs/research/agent-native-processes.md` (roadmap item P8).
- **New skill `/flywheel:process`** (`skills/process/SKILL.md`) — scaffolds a
  **process contract** at `.claude/flywheel/processes/<slug>.md`: fixed rules, an
  output schema, a persistence target, bounded judgment latitude, and an
  append-only improvement log. On first use it bootstraps `.claude/flywheel/DATA.md`
  by detecting the repo's existing persistence strategy (ORM configs,
  `DATABASE_URL`, `supabase/`, migrations, compose services) before asking.
- **New skill `/flywheel:run`** (`skills/run/SKILL.md`) — executes a contract as
  the runtime: follow the rules, apply judgment only where allowed, emit the
  output schema, **persist to the repo's datastore per `DATA.md`** (idempotent,
  transactional, read-back verified — the same "observe it land" rigor
  `/flywheel:work` holds for tests), optionally cross-check with the `evaluator`
  agent when the process declares a metric, then append at most one
  evidence-based refinement to the contract's improvement log.
- **New repo state** (in the project flywheel is used on):
  `.claude/flywheel/processes/<slug>.md` and `.claude/flywheel/DATA.md`. Same
  home as `specs/` and `LEARNINGS.md`; nothing pre-existing is touched.
- **Docs:** README gains a "second pillar" section, both commands in the command
  table, and the two new state files; `/flywheel:help` documents the pillar and
  both commands; the SessionStart banner lists them; new root `CLAUDE.md` records
  the north-star.
- **No config or ledger-format changes; no changes to existing commands.** The
  vendoring installer auto-discovers skills, so re-vendoring a web repo picks up
  `/flywheel-process` and `/flywheel-run` with no installer edit; marketplace
  installs get them on the next `/plugin update` + reload. Nothing to migrate in
  installed repos — hence `requires-action: false`.
